## API 开发规范

### 1. 标准 API 接口模板

所有 API 接口必须遵循以下开发模式：

#### 核心原则

1. **强制鉴权**：搜索、详情、用户相关接口必须使用 `withAuth`
2. **零侵入监控**：使用 `withMonitoring` 或 `withAuthMonitoring` 包装 handler
3. **请求方式判断**：在最外层统一判断 HTTP 方法
4. **标准化响应**：使用 `responseHelper` 的标准函数
5. **数据库操作分离**：禁止在 API 接口文件中直接操作数据库

#### 标准模板示例

```typescript
// src/pages/api/example.ts
import { NextApiRequest, NextApiResponse } from "next";
import { withAuth } from "@/middleware/auth/withAuth";
import { withAuthMonitoring } from "@/utils/withMonitoring";
import {
  sendSuccessResponse,
  sendWarnningResponse,
  sendErrorResponse,
  sendMethodNotAllowedResponse,
} from "@/helper/responseHelper";
import logRequest from "@/utils/logRequest";

/**
 * 业务逻辑处理函数
 */
const handlePost = async (
  req: NextApiRequest,
  res: NextApiResponse,
  userId: string
) => {
  const { param1, param2 } = req.body;

  // 参数验证
  if (!param1) {
    return sendWarnningResponse(res, "参数不能为空");
  }

  // 调用 db 层函数（不直接操作数据库）
  const result = await someDbFunction(param1, param2);

  return sendSuccessResponse(res, "操作成功", result);
};

/**
 * 请求方式判断（最外层）
 */
const handler = async (
  req: NextApiRequest,
  res: NextApiResponse,
  userId: string
) => {
  try {
    if (req.method === "POST") {
      return await handlePost(req, res, userId);
    } else {
      return sendMethodNotAllowedResponse(res, "仅支持POST请求");
    }
  } catch (error) {
    console.error("操作失败:", error);
    return sendErrorResponse(res, "操作失败，请稍后再试");
  }
};

/**
 * 导出 - 鉴权 + 监控
 */
export default withAuth(
  withAuthMonitoring(handler, {
    monitorType: "business",
    operationName: "example_operation",
    extractMetadata: (req) => ({ param1: req.body.param1 }),
    successMetric: "example_success",
    failureMetric: "example_failed",
  })
);
```

### 2. 响应格式规范

**禁止直接使用 `res.status().json()`，必须使用 `responseHelper` 函数：**

#### 可用的响应函数

```typescript
import {
  sendSuccessResponse, // 200 - 成功
  sendErrorResponse, // 501 - 业务错误
  sendWarnningResponse, // 503 - 业务警告
  sendUnauthorizedResponse, // 401 - 未授权
  sendMethodNotAllowedResponse, // 405 - 方法不允许
} from "@/helper/responseHelper";
```

#### 使用示例

```typescript
// 正确：使用标准响应函数
sendSuccessResponse(res, "操作成功", { id: "123" });
sendWarnningResponse(res, "参数不能为空");
sendErrorResponse(res, "操作失败");

// 错误：禁止直接使用 res.status().json()
res.status(200).json({ code: 200, message: "成功", data: {...} }); // ❌
res.status(400).json({ code: 400, message: "失败" }); // ❌
```

#### 响应格式说明

所有响应遵循统一格式：

```typescript
interface ApiResponse<T = any> {
  code: number; // HTTP 状态码：200 成功，501 错误，503 警告
  message: string; // 提示信息
  data: T; // 响应数据
}
```

### 3. 监控类型选择

| 监控类型       | 使用场景                                | 记录内容                     |
| -------------- | --------------------------------------- | ---------------------------- |
| `external_api` | 调用第三方 API（AMiner、OpenAI、EBSCO） | API 响应时间、结果数量、错误 |
| `business`     | 业务逻辑操作（登录、注册、收藏）        | 操作结果、耗时、用户行为     |
| `none`         | 简单接口（仅依赖 logRequest）           | 仅记录基础请求日志           |

## 前端开发规范

### 1. HTTP 请求规范

**所有前端 HTTP 请求必须使用统一封装的 API 方法：**

#### 统一请求方法

```typescript
import { apiGet, apiPost, apiPut, apiDel } from "@/api/request";

// GET 请求
const response = await apiGet<UserData>("/api/user/profile");

// POST 请求（带数据）
const response = await apiPost<LoginResponse>("/api/auth/login", {
  phone_number: "13800138000",
  password: "password123",
});

// PUT 请求（更新数据）
const response = await apiPut<UpdateResponse>("/api/user/profile", {
  nickname: "新昵称",
});

// DELETE 请求
const response = await apiDel<DeleteResponse>("/api/user/favorites/123");
```

#### 禁止事项

```typescript
// ❌ 禁止直接使用 axios
import axios from "axios";
axios.post("/api/auth/login", data);

// ❌ 禁止直接使用 fetch（除非有特殊需求如 SSE）
fetch("/api/auth/login", { method: "POST", body: JSON.stringify(data) });
```

#### 错误处理

封装会自动处理常见错误，业务代码只需捕获特定业务逻辑错误：

```typescript
try {
  const response = await apiPost("/api/auth/login", formData);
  console.log("登录成功:", response.data);
} catch (error: any) {
  // 错误已被统一处理（toast 已显示），这里只需显示错误信息
  setError(error.message || "登录失败");
}
```

## 数据库操作规范

### 1. 数据库操作分离原则

**禁止在 API 接口文件中直接进行数据库操作！**

#### 规范说明

1. **API 层职责**（`src/pages/api/`）：

   - ✅ 参数验证
   - ✅ 调用 service 层或 db 层函数
   - ✅ 返回标准化响应
   - ❌ 不应包含任何 `prisma.xxx.findMany()` 等数据库操作

2. **数据库层职责**（`src/db/`）：

   - ✅ 封装所有 Prisma 数据库操作
   - ✅ 提供类型安全的函数接口
   - ✅ 处理数据库事务和错误

3. **Service 层职责**（`src/service/`）：
   - ✅ 调用第三方 API
   - ✅ 业务逻辑编排
   - ✅ 数据格式转换

#### 错误示例

```typescript
// ❌ 错误：直接在 API 中操作数据库
const handler = async (req, res, userId) => {
  const dbPapers = await prisma.paper.findMany({
    where: { source: "aminer" },
  });

  const favoriteMap = await prisma.favorite.findMany({
    where: { user_id: userId },
  });
};
```

#### 正确示例

```typescript
// ✅ 正确：调用 db 层封装的函数

// src/db/aminer/paper.ts
export async function findPapersBySource(source: string) {
  return await prisma.paper.findMany({
    where: { source },
  });
}

// src/pages/api/search/papers.ts
const handler = async (req, res, userId) => {
  const dbPapers = await findPapersBySource("aminer");
  const favoriteMap = await batchCheckFavorites({
    user_id: userId,
    favorite_type: "paper",
    item_ids: paperIds,
  });
};
```

### 2. 数据库迁移规范

**禁止自动执行数据库迁移！**

#### 规范流程

1. **修改 schema**：在 `prisma/schema.prisma` 中修改模型定义
2. **生成 Prisma Client**：
   ```bash
   pnpm prisma generate
   ```
3. **创建迁移 SQL**：

   - 在 `prisma/migrations/` 目录下创建新的迁移文件夹（格式：`YYYYMMDDHHMMSS_描述`）
   - 在文件夹中创建 `migration.sql` 文件，编写 SQL 语句
   - **禁止使用 `pnpm prisma migrate dev`**
   - **禁止使用 `pnpm prisma db push`**

4. **提交 SQL 给用户**：
   - 将 SQL 文件路径告知用户
   - 用户手动审核后在数据库中执行

#### 示例迁移 SQL

```sql
-- 文件：prisma/migrations/20251123225510_add_reasoning_fields_to_bill/migration.sql

-- AlterTable
ALTER TABLE `bills`
  ADD COLUMN `reasoning_content` TEXT NULL COMMENT 'DeepSeek-R1 思考过程内容',
  ADD COLUMN `reasoning_tokens` INT NULL DEFAULT 0 COMMENT '思考过程消耗的 tokens';
```

#### 安全检查清单

- [ ] SQL 语句已创建在 `prisma/migrations/` 目录
- [ ] 已执行 `pnpm prisma generate` 生成类型
- [ ] 已告知用户 SQL 文件路径
- [ ] **未执行** `prisma migrate dev` 或 `prisma db push`

## 命名规范

### 1. API 参数命名规范

**所有 API 入参和出参必须采用下划线形式命名（snake_case）**

#### 规范说明

1. **请求参数命名**：所有从客户端传入的参数使用下划线命名

   - ✅ **正确**: `conversation_id`, `deep_thinking`, `cited_paper_ids`, `search_type`
   - ❌ **错误**: `conversationId`, `deepThinking`, `citedPaperIds`, `searchType`

2. **响应字段命名**：返回给客户端的数据字段也应使用下划线命名

   - ✅ **正确**: `user_id`, `create_time`, `is_favorited`, `paper_id`
   - ❌ **错误**: `userId`, `createdAt`, `isFavorited`, `paperId`

3. **数据库字段映射**：与 Prisma 数据库字段命名保持一致
   - Prisma schema 使用下划线命名，API 参数应直接对应

#### 示例对比

```typescript
// ❌ 错误：驼峰命名
const { conversationId, deepThinking, citedPaperIds } = req.body;

// ✅ 正确：下划线命名
const { conversation_id, deep_thinking, cited_paper_ids } = req.body;
```

#### 特殊场景

- **环境变量**：使用全大写 + 下划线，如 `LLM_MODEL`, `DATABASE_URL`
- **TypeScript 类型定义**：接口字段使用下划线命名，保持与 API 一致
- **前端传参**：前端发送请求时也应使用下划线命名，确保前后端一致

### 2. 密码编码规范

**前端传递密码时必须进行 base64 编码：**

```typescript
// 正确：使用 btoa() 进行 base64 编码
const encodedPassword = btoa(password);

const response = await apiPost("/api/auth/login", {
  phone_number: "13800138000",
  password: btoa("a123456"),
});

// 错误：直接传递明文密码
const response = await apiPost("/api/auth/login", {
  phone_number: "13800138000",
  password: "a123456", // ❌
});
```

### 3. 文件命名规范

- **组件文件**：PascalCase，如 `UserProfile.tsx`, `LoginForm.tsx`
- **工具函数**：camelCase，如 `formatDate.ts`, `validateEmail.ts`
- **API 路由**：kebab-case，如 `reset-pwd.ts`
- **hook**：kebab-case，如 `use-auth.ts`
- **数据库层**：camelCase，如 `user.ts`, `paper.ts`

## UI 组件规范

### 1. 样式类名合并规范

**使用 `cn` 函数合并 Tailwind CSS 类名：**

项目中已集成 `tailwind-merge` 和 `clsx`，提供了 `cn` 工具函数来智能合并类名，避免类名冲突。

#### cn 函数定义

```typescript
// src/lib/utils.ts
import { clsx, type ClassValue } from "clsx";
import { twMerge } from "tailwind-merge";

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}
```

#### 使用场景

**场景 1：组件基础样式 + 条件样式 + 自定义样式**

```typescript
import { cn } from "@/lib/utils";

export const AuthInput = React.forwardRef<HTMLInputElement, AuthInputProps>(
  ({ error, className, ...props }, ref) => {
    return (
      <input
        ref={ref}
        className={cn(
          // 基础样式
          "w-[360px] h-[40px] px-4 py-2 border rounded-[10px]",
          "focus:outline-none focus:ring-2 transition-all text-base bg-white",
          // 条件样式
          error
            ? "border-red-500 focus:ring-red-500"
            : "border-[#C8C9CC] focus:ring-blue-500",
          // 外部传入的自定义样式
          className
        )}
        {...props}
      />
    );
  }
);
```

**场景 2：多个条件样式组合**

```typescript
<div
  className={cn(
    "flex items-center gap-2",
    isActive && "bg-blue-500 text-white",
    isDisabled && "opacity-50 cursor-not-allowed",
    size === "lg" && "h-12 text-lg",
    size === "sm" && "h-8 text-sm",
    className
  )}
>
  {children}
</div>
```

**场景 3：数组形式的类名**

```typescript
const buttonStyles = cn([
  "px-4 py-2 rounded-lg",
  "transition-all duration-200",
  isPrimary ? "bg-blue-500 hover:bg-blue-600" : "bg-gray-200 hover:bg-gray-300",
]);
```

#### 为什么使用 cn 函数？

| 方式                                              | 问题                            | cn 函数解决 |
| ------------------------------------------------- | ------------------------------- | ----------- |
| **字符串拼接** `${baseClass} ${conditionalClass}` | 冗长、易错、类名冲突            | ❌ 不推荐   |
| **模板字符串** `\`w-4 ${isLarge && 'w-8'}\``      | 类名冲突（w-4 和 w-8 同时存在） | ❌ 不推荐   |
| **cn 函数** `cn("w-4", isLarge && "w-8")`         | 自动去重，后者覆盖前者          | ✅ 推荐     |

**示例对比：**

```typescript
// 错误：直接拼接，w-4 和 w-8 同时存在，渲染结果不可预测
<div className={`w-4 ${isLarge && 'w-8'}`} />
// 实际输出：class="w-4 w-8" （冲突）

// 正确：使用 cn 函数，自动去重
<div className={cn("w-4", isLarge && "w-8")} />
// 实际输出：class="w-8" （w-8 覆盖 w-4）
```

#### 规范要求

1. **禁止直接拼接类名**

   ```typescript
   // ❌ 错误
   className={`${baseClass} ${className}`}
   className={baseClass + " " + className}

   // ✅ 正确
   className={cn(baseClass, className)}
   ```

2. **条件类名统一使用 cn 函数**

   ```typescript
   // ❌ 错误
   className={isActive ? "active-class" : "inactive-class"}

   // ✅ 正确
   className={cn(
     "base-class",
     isActive ? "active-class" : "inactive-class"
   )}
   ```

3. **组件接受外部 className 时必须使用 cn 合并**

   ```typescript
   // ❌ 错误：外部样式可能被内部样式覆盖
   <div className={`flex items-center ${className}`} />

   // ✅ 正确：cn 函数智能合并，外部样式优先级更高
   <div className={cn("flex items-center", className)} />
   ```

4. **能用 shadcn/ui 组件的，必须使用 shadcn/ui 组件实现**

开发前端界面时，应当优化考虑当前功能&需求是否能使用 shadcn/ui 组件实现，若能，必须使用 shadcn/ui 组件实现。

### 2. shadcn/ui 组件规范

- 使用 `pnpm dlx shadcn@latest add <component>` 安装组件
- 保持组件与官方文档一致
- 自定义样式应通过 Tailwind CSS 类名实现
- 颜色系统使用 CSS 变量（`src/styles/global.css`）

## 错误处理规范

### 1. API 响应拦截器

前端 axios 拦截器自动处理以下错误：

- **401 未授权**：自动清除 Token 并跳转到登录页（携带回调参数）
- **403 无权限**：提示"没有权限访问"
- **404 不存在**：提示"请求的资源不存在"
- **500+ 服务器错误**：提示"服务器错误"
- **网络错误**：提示"网络错误，请检查网络连接"

### 2. 后端错误处理

使用 try-catch 捕获错误，并使用 `sendErrorResponse` 返回：

```typescript
try {
  const result = await someOperation();
  return sendSuccessResponse(res, "操作成功", result);
} catch (error) {
  console.error("操作失败:", error);
  return sendErrorResponse(res, "操作失败，请稍后再试");
}
```

### 3. 业务错误对象

前端错误对象类型：

```typescript
interface BusinessError {
  message: string;
  isWarning: boolean;
  code?: number;
}
```

## 文件组织规范

### 1. 文件大小限制

**⚠️ 每个文件的代码行数不应超过 500 行！**

**规范说明：**

1. **单文件行数限制**：

   - ✅ **最大行数**：500 行（包括空行和注释）
   - ⚠️ **建议行数**：0-350 行
   - ❌ **禁止**：单文件超过 500 行

2. **超过限制时的处理**：
   - **拆分组件**：将大组件拆分为多个小组件
   - **提取逻辑**：将复杂逻辑提取为独立的 hooks 或 utils 函数
   - **步骤拆分**：将多步骤流程拆分为独立的步骤组件
   - **模块化**：按功能模块划分子目录

**示例 ✅：**

```typescript
// ❌ 错误：单文件 900+ 行
// src/components/auth/ForgotPasswordPage.tsx

// ✅ 正确：拆分为多个文件
// src/components/auth/forgot-password/
//   ├── ForgotPasswordPage.tsx (104 lines)
//   ├── ForgotPasswordStep1.tsx (89 lines)
//   ├── ForgotPasswordStep2.tsx (118 lines)
//   └── ForgotPasswordStep3.tsx (105 lines)
```

**拆分原则：**

- **单一职责**：每个文件只负责一个功能
- **可测试性**：小文件更容易编写单元测试
- **可维护性**：降低认知负担，提高可读性
- **可复用性**：拆分后的组件可以在其他地方复用

### 2. 导入路径规范

**⚠️ 除同级或子级文件外，所有导入必须使用 @ 别名！**

**规范说明：**

1. **使用 @ 别名的场景**：

   - ✅ **跨目录导入**：导入其他目录的模块
   - ✅ **父级目录导入**：导入上层目录的模块
   - ✅ **工具函数导入**：导入 utils、hooks、types 等
   - ✅ **第三方库配置导入**：导入 lib 目录的配置

2. **使用相对路径的场景**：
   - ✅ **同级文件**：`import { Component } from "./Component"`
   - ✅ **子级文件**：`import { Helper } from "./utils/helper"`
   - ❌ **父级文件**：不使用 `../`

**正确示例 ✅：**

```typescript
// src/components/auth/forgot-password/ForgotPasswordPage.tsx

// ✅ 同级文件 - 使用相对路径
import { ForgotPasswordStep1 } from "./ForgotPasswordStep1";

// ✅ 子级文件（layout 是同级目录的子项）- 使用相对路径
import { AuthPageLayout } from "../layout/AuthPageLayout";

// ✅ 跨目录（types）- 使用 @ 别名
import type { RegisterStep1Data } from "@/types/auth";

// ✅ hooks - 使用 @ 别名
import { useFormValidation } from "@/hooks/use-form-validation";

// ✅ 工具函数 - 使用 @ 别名
import { cn } from "@/lib/utils";

// ✅ 组件（跨模块）- 使用 @ 别名
import { AuthInput, AuthButton } from "@/components/auth";
```

**错误示例 ❌：**

```typescript
// ❌ 错误：跨目录使用相对路径
import { useAuth } from "../../hooks/use-auth";

// ✅ 正确：使用 @ 别名
import { useAuth } from "@/hooks/use-auth";

// ❌ 错误：父级目录使用相对路径
import { AuthPageLayout } from "../../../components/auth/layout/AuthPageLayout";

// ✅ 正确：使用 @ 别名
import { AuthPageLayout } from "@/components/auth/layout/AuthPageLayout";
```

**路径别名配置：**

项目已配置以下别名（`tsconfig.json`）：

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

**优势：**

- **可维护性**：文件移动时不需要修改导入路径
- **可读性**：清晰地表明导入的来源
- **一致性**：统一的导入风格
- **重构友好**：IDE 支持自动重构
